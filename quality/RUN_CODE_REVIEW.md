## Bootstrap
Read these artifacts before running Pass 1:
1. `quality/QUALITY.md`
2. `quality/REQUIREMENTS.md`
3. `quality/CONTRACTS.md`
4. `quality/COVERAGE_MATRIX.md`
5. `quality/EXPLORATION.md`

## Pass 1: Structural Review
### Mandatory scrutiny sections
1. Input validation and boundary handling
2. Resource lifecycle
3. Concurrency and state management
4. Unit and encoding correctness
5. Enumeration and whitelist completeness

### Guardrails
- Line numbers are mandatory.
- Read full function bodies before raising a finding.
- Use `QUESTION` when uncertain, not `BUG`.
- Grep before claiming absence.
- Exclude style-only suggestions.

### Enumeration completeness mechanics
- Use existing `quality/mechanical/*_cases.txt` artifacts when available.
- If an artifact is missing, extract code-side lists with line-numbered entries before claiming completeness.
- Compare code-side list to spec/contract-side list and report diffs explicitly.

## Pass 2: Requirement Verification
For every REQ in `quality/REQUIREMENTS.md`:
- Assign one verdict: `SATISFIED`, `VIOLATED`, `PARTIALLY SATISFIED`, or `NOT ASSESSABLE`.
- Provide at least one code citation per REQ.
- For REQs with per-site UCs (`UC-2.a`, `UC-2.b`), verify each site separately.
- For pattern-tagged REQs (`REQ-001`..`REQ-004`), do not skip evidence; include route/function-specific proof.

## Pass 3: Cross-Requirement Consistency
Compare REQ pairs sharing data contracts or policy boundaries:
- REQ-001 vs REQ-002: fallback behavior vs downstream envelope assumptions.
- REQ-002 vs REQ-005: route contract consistency vs runtime pull degradation semantics.
- REQ-003 vs REQ-010/REQ-011: telemetry vocabulary consistency vs review/audit artifact interpretation.
- REQ-004 vs REQ-013: approval gates vs mechanical verification of allowlists and dispatch boundaries.

## Output Format
Create one review artifact in `quality/code_reviews/` with sections:
- `## Pass 1: Structural Review`
- `## Pass 2: Requirement Verification`
- `## Pass 3: Cross-Requirement Consistency`
- `## Combined Summary`

Each finding entry format:
- `Source: <pass>`
- `Requirement: <REQ-NNN or N/A>`
- `Location: <path:line>`
- `Verdict: BUG | QUESTION | VIOLATED | PARTIALLY SATISFIED | INCONSISTENT`
- `Impact: HIGH | MEDIUM | LOW`
- `Evidence: <quoted or paraphrased code behavior>`
- `Regression test linkage: <test name or exemption>`

## Closure Rules
- Every confirmed BUG must map to a regression test in `quality/test_regression.*` or an explicit exemption.
- For every requirement violation, include expected vs actual behavior.
- If a finding is overturned during spec audit triage, remove/reclassify its regression test.
