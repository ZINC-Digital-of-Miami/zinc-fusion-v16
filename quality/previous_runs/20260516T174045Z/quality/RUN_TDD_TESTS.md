# TDD Verification Protocol: ZINC-FUSION-V16
> Generated by [Quality Playbook](https://github.com/andrewstellman/quality-playbook) v1.5.6 — Andrew Stellman
> Date: 2026-05-16 · Project: ZINC-FUSION-V16

## Scope

Run red/green verification for every confirmed bug in `quality/BUGS.md`.

## Spec-grounded test requirements

For each BUG-NNN:

1. Read associated requirement (`REQ-NNN`) from `quality/REQUIREMENTS.md`.
2. Read bug `spec basis` field.
3. Build assertions using spec language (not internal-only shorthand).
4. Add traceability comment with requirement ID, bug ID, and spec passage.

## Red-Green Execution

For each bug with regression and fix patches:

1. **Red phase:** run regression test on unpatched code. Expected: fail.
2. **Green phase:** apply fix patch and rerun same test. Expected: pass.
3. **Record:** write logs and update closure status.

If no fix patch exists:
- Red phase still required.
- Green phase marked `skipped`.
- Verdict is `confirmed open`.

## Framework adaptation

Project baseline test framework: Python `unittest`.

Primary command pattern:

```bash
python -m unittest quality.test_regression
```

For per-bug runs:

```bash
python -m unittest quality.test_regression.TestRegression.test_bug_nnn
```

## Required Log Artifacts

For each confirmed BUG-NNN:

- `quality/results/BUG-NNN.red.log`
- `quality/results/BUG-NNN.green.log` (if fix patch exists)

First line must be one of: `RED`, `GREEN`, `NOT_RUN`, `ERROR`.

## Log Template

```text
RED
--- Test output for BUG-NNN red phase ---
Command: python -m unittest quality.test_regression.TestRegression.test_bug_nnn
Exit code: 1
[stdout/stderr]
```

## Traceability Output

Generate `quality/TDD_TRACEABILITY.md`:

| Bug ID | Requirement ID | Spec Doc | Spec Section | Behavioral Contract | Test File:Function | Red Result | Green Result |
|---|---|---|---|---|---|---|---|

Every confirmed bug must have a row.

## Patch Gate

Before bug closure:

1. `git apply --check quality/patches/BUG-NNN-regression-test.patch`
2. `git apply --check quality/patches/BUG-NNN-fix.patch` (if present)
3. Run syntax/compile check after applying patches in temporary worktree.

If patch gate fails, repair patch before claiming closure.

## Execution UX

### Phase 1: Plan

Show bug verification queue as numbered table.

### Phase 2: Progress

One-line progress per bug:
- `FAIL ✓ -> PASS ✓`
- `FAIL ✓ -> SKIPPED (no fix patch)`
- `ERROR (environment)`

### Phase 3: Results

Report totals:
- Verified
- Confirmed open
- Red failed
- Green failed
- Deferred

## Upstream-ready report block

For each `TDD verified` bug include:

1. One-sentence spec violation summary.
2. FAIL→PASS terminal output snippet.
3. Regression test path.
4. Fix patch path.
