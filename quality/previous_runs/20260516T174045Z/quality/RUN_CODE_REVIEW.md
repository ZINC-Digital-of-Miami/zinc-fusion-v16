## Pass 1: Structural Review

Review code for correctness defects only. No style feedback.

### Mandatory guardrails

- Line numbers required for every finding.
- Read function bodies, not signatures only.
- Use `QUESTION` if confidence is insufficient.
- Grep before claiming absence.
- No refactor/style findings.

### Required scrutiny areas

1. Input validation and trust boundaries (API envelopes, whitelist gates, fallback routing).
2. Resource lifecycle (ingest run creation/update and terminal states).
3. Concurrency/state integrity (run lifecycle transitions, status updates, stale-metadata propagation).
4. Unit/encoding correctness (date semantics, horizon values, envelope field types).
5. Enumeration/whitelist completeness with mechanical two-list diff.

## Pass 2: Requirement Verification

Evaluate each REQ individually using `quality/REQUIREMENTS.md`.

For each REQ, output:

- `Status:` SATISFIED / VIOLATED / PARTIALLY SATISFIED / NOT ASSESSABLE
- `Evidence:` file:line references
- `Analysis:` expected vs observed behavior
- `Severity:` required for VIOLATED/PARTIALLY SATISFIED

### REQ verification checklist

- REQ-001: Confirm all ingest writers satisfy unified `run_id` + status contract.
- REQ-002: Confirm no middleware/gate dependency on `/api/cron`.
- REQ-003: Confirm intraday fallback behavior and contract text alignment.
- REQ-004: Confirm envelope contract parity across types/routes/gates.
- REQ-005: Confirm mixed-vintage freshness semantics are explicit and non-misleading.
- REQ-006: Confirm identical normalization logic across all forecast surfaces.

## Pass 3: Cross-Requirement Consistency

Check shared concepts across REQ pairs.

Required pairings:

- REQ-001 vs REQ-004: shared ingest/API contract normalization principles.
- REQ-002 vs REQ-004: gate policy consistency with architecture authority.
- REQ-004 vs REQ-006: forecast surface consistency (envelope + horizon semantics).
- REQ-003 vs REQ-005: degradation/fallback communication consistency.

For each pair, report CONSISTENT or INCONSISTENT with file:line evidence.

## Mechanical Enumeration Rules

If a finding claims whitelist/dispatch completeness, reviewer MUST:

1. Use `quality/mechanical/*` extraction artifacts when available.
2. Compare extracted code list vs authoritative list.
3. Cite exact case/entry lines.
4. Mark any missing entry as BUG candidate.

## Regression Test Closure Mandate

Every confirmed BUG requires one of:

- A regression test in `quality/test_regression.*` with expected-failure guard.
- An explicit exemption rationale in review output.

Patch artifacts required per confirmed bug:

- `quality/patches/BUG-NNN-regression-test.patch` (mandatory)
- `quality/patches/BUG-NNN-fix.patch` (strongly encouraged)
