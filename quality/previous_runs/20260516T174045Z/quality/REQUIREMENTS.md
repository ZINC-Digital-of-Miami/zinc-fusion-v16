## Project Overview

ZINC-FUSION-V16 is a commodity procurement forecasting platform focused on ZL futures price-level intelligence for buy-side operations. The system combines Supabase-hosted ingestion, Next.js API and UI surfaces, and a Python forecasting pipeline. Operational safety depends on contract alignment across SQL ingest writers, API envelopes, horizon normalization, and freshness semantics presented to procurement users.

This requirements set is derived from Phase 1 exploration evidence in `quality/EXPLORATION.md` and targets the highest-risk divergence vectors discovered there: telemetry contract split, architecture/gate drift, fallback-contract mismatch, envelope inconsistency, mixed-vintage miscommunication, and duplicated horizon logic.

## Use Cases

### UC-01: Record an ingest run lifecycle event across SQL and TS writers

- Actors: pg_cron SQL functions, API helper writer.
- Preconditions: An ingest job starts, succeeds, or fails.
- Steps:
  1. Writer allocates or receives a run identifier.
  2. Writer emits start record to `ops.ingest_run`.
  3. Writer updates same run record to terminal status and timestamps.
- Postconditions: Telemetry rows are query-consistent across writers.
- Alternative paths: On ingest failure, terminal status and error message are still written.
- Requirements: REQ-001

### UC-02: Verify architecture-compliant auth and gate behavior without cron-route dependency

- Actors: Middleware, verification scripts.
- Preconditions: Protected route request or verification run.
- Steps:
  1. Middleware enforces auth/session rules.
  2. Verification gates evaluate architecture compliance.
  3. No component assumes `/api/cron` surface exists.
- Postconditions: Compliant builds are not blocked by forbidden-route checks.
- Alternative paths: Gate outputs actionable architecture mismatch evidence.
- Requirements: REQ-002

### UC-03: Serve intraday ZL bars with deterministic fallback semantics

- Actors: `/api/zl/intraday` route.
- Preconditions: User requests intraday bars.
- Steps:
  1. Route loads primary 15m dataset.
  2. If primary unavailable, route follows explicitly documented fallback behavior.
  3. Response contract communicates the data-source path used.
- Postconditions: Client expectations match real behavior.
- Alternative paths: If no fallback exists, route and docs declare no-fallback explicitly.
- Requirements: REQ-003

### UC-04: Enforce consistent envelope contract across forecast and dashboard APIs

- Actors: API routes, contract types, parity gate scripts.
- Preconditions: Success or failure responses emitted by covered endpoints.
- Steps:
  1. Contract type declares required and optional fields.
  2. Route handlers serialize envelopes accordingly.
  3. Gate checks validate only declared requirements.
- Postconditions: Parity failures indicate product defects, not harness drift.
- Alternative paths: Endpoint-specific exceptions are documented and consistently enforced.
- Requirements: REQ-004

### UC-05: Render risk-factor freshness with transparent mixed-vintage semantics

- Actors: Risk-factor API and dashboard card UI.
- Preconditions: Component timestamps differ across factors.
- Steps:
  1. API emits min/max/as-of and mixed-vintage fields.
  2. UI shows degraded-state freshness messaging.
  3. Operator can distinguish globally stale from partially stale vectors.
- Postconditions: Procurement timing decisions use correct recency confidence.
- Alternative paths: If metadata absent, UI displays explicit unknown-freshness state.
- Requirements: REQ-005

### UC-06.a: Normalize horizon values in `/api/zl/forecast`

- Actors: Forecast route.
- Preconditions: Rows include `horizon_days` and `model_version`.
- Steps: Apply canonical allowlist and legacy remap.
- Postconditions: Returned series uses canonical horizon set.
- Requirements: REQ-006

### UC-06.b: Normalize horizon values in `/api/zl/target-zones`

- Actors: Target-zones route.
- Preconditions: Rows include `horizon_days` and `model_version`.
- Steps: Apply same normalization logic as UC-06.a.
- Postconditions: Target-zone horizons align with forecast horizons.
- Requirements: REQ-006

### UC-06.c: Normalize horizon values in `/api/zl/forecast-targets`

- Actors: Forecast-targets route.
- Preconditions: Rows include `horizon_days` and `model_version`.
- Steps: Apply same normalization logic as UC-06.a and UC-06.b.
- Postconditions: Probability-surface horizons align with other forecast surfaces.
- Requirements: REQ-006

## Cross-Cutting Concerns

- **Architecture authority:** Repository policy (`AGENTS.md`) is authoritative over historical middleware or gate assumptions.
- **Telemetry integrity:** Ingest observability must stay coherent across language boundaries (SQL and TypeScript writers).
- **Contract traceability:** API contract types, route payloads, and gate scripts must remain synchronized.
- **Degradation transparency:** Fallback and mixed-vintage behaviors must be explicit in machine and user-facing surfaces.

## Functional Sections

### Ingestion Observability and Lifecycle Contracts

#### REQ-001: `ops.ingest_run` writers MUST use one normalized run contract

- Functional section: Ingestion Observability and Lifecycle Contracts
- Tier: 3
- Pattern: parity
- References: `supabase/migrations/202603180009_ops_vegas.sql:3-14`; `supabase/migrations/20260414001_ingest_zl_intraday.sql:50-52`; `supabase/migrations/202605080003_supabase_trusted_site_fill_hourly.sql:55-57`; `lib/server/ingest-run.ts:3-4`
- Linked use cases: UC-01
- Requirement: Every SQL and TypeScript writer that inserts or updates `ops.ingest_run` MUST include stable `run_id` lifecycle semantics and one canonical status vocabulary.
- Conditions of satisfaction:
  1. `ops.ingest_run` DDL defines `run_id UUID PRIMARY KEY` and writer inserts honor that key.
  2. Each writer emits start and terminal state for the same `run_id`.
  3. Status values are normalized (no mixed uppercase/lowercase contract split).
  4. Error paths log terminal status with timestamp and error detail.

### Architecture Policy and Verification Alignment

#### REQ-002: Verification and auth scaffolding MUST not require `/api/cron`

- Functional section: Architecture Policy and Verification Alignment
- Tier: 3
- Pattern: whitelist
- References: `AGENTS.md:44-45`; `AGENTS.md:149-160`; `lib/supabase/proxy.ts:15-42`; `scripts/verify/gate3.sh:5`
- Linked use cases: UC-02

### Intraday Data Contract Fidelity

#### REQ-003: `/api/zl/intraday` MUST align documented fallback with implementation

- Functional section: Intraday Data Contract Fidelity
- Tier: 3
- References: `app/api/zl/intraday/route.ts:10-16`; `supabase/migrations/202603180002_mkt.sql:17-20`
- Linked use cases: UC-03
- Requirement: Intraday route behavior, comments, and consumer assumptions MUST agree on whether 1m fallback exists.
- Conditions of satisfaction:
  1. If fallback is claimed, route executes an explicit 15m→1m fallback query path.
  2. If fallback is not implemented, comments and downstream expectations state that explicitly.
  3. Response metadata identifies the effective data source path for debugging and parity checks.

### API Envelope and Parity Contract Consistency

#### REQ-004: Forecast/dashboard API surfaces, contract types, and parity harness MUST agree on envelope semantics

- Functional section: API Envelope and Parity Contract Consistency
- Tier: 3
- Pattern: parity
- References: `lib/contracts/api.ts:3-9`; `scripts/verify/gate6.sh:74-80`; `app/api/zl/live/route.ts:28-33`; `app/api/zl/target-zones/route.ts:74-79`; `app/api/zl/forecast-targets/route.ts:97-100`

### Freshness Transparency for Procurement Decisions

#### REQ-005: Risk-factor freshness MUST expose mixed-vintage state without mislabeling oldest timestamp as global freshness

- Functional section: Freshness Transparency for Procurement Decisions
- Tier: 3
- Pattern: compensation
- References: `app/api/dashboard/risk-factors/route.ts:715-719`; `app/api/dashboard/risk-factors/route.ts:815-819`; `components/dashboard/MarketRiskFactors.tsx:64-77`; `components/dashboard/MarketRiskFactors.tsx:357-358`
- Linked use cases: UC-05
- Requirement: API and UI freshness semantics MUST make degraded recency explicit when component dates diverge.
- Conditions of satisfaction:

  2. UI copy distinguishes mixed-vintage from globally stale data.
  3. Freshness labels do not imply full-vector recency when only subset is stale.

### Forecast Horizon Normalization Consistency

#### REQ-006: Horizon normalization MUST remain identical across all forecast-facing ZL endpoints

- Functional section: Forecast Horizon Normalization Consistency
- Tier: 3
- Pattern: parity
- References: `app/api/zl/forecast/route.ts:13-24`; `app/api/zl/target-zones/route.ts:13-24`; `app/api/zl/forecast-targets/route.ts:22-33`
- Linked use cases: UC-06.a, UC-06.b, UC-06.c
- Requirement: `normalizeHorizon` allowlist and legacy remap behavior MUST remain equivalent for forecast, target-zones, and forecast-targets endpoints.
- Conditions of satisfaction:
  1. Each endpoint uses identical `AG_HORIZON_DAYS` and `LEGACY_HORIZON_MAP` definitions.
  2. Legacy model versions apply the same remap (`7→30`, `14→90`, `30→180`).
  3. Unknown horizons are filtered consistently and logged consistently.
