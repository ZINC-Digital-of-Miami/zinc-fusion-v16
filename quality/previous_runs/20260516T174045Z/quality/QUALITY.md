## Purpose

Quality for ZINC-FUSION-V16 means procurement decisions are safe to act on under real market volatility, not just that routes return HTTP 200. Data freshness, contract consistency, and horizon alignment must stay correct across ingest SQL, API handlers, and dashboard rendering.

Deming applies here as an engineering workflow rule: quality is built into artifacts that every future AI or human session must inherit (`QUALITY.md`, `REQUIREMENTS.md`, protocol docs, and functional tests). Juran applies as fitness-for-use: Chris and Kevin need reliable ZL price-level intelligence to make buy/wait decisions. Crosby applies as operational cost control: one hour spent preventing silent freshness drift is cheaper than a week of bad procurement timing.

## Coverage Theater Prevention

- Passing tests that only assert `ok: true` without validating envelope shape.
- Testing fallback code by asserting output exists, but not asserting downgrade indicators.
- Source-inspection tests that never run (`run=False`, disabled guards left in place forever).
- Tests that verify only one forecast endpoint while claiming horizon consistency for three endpoints.
- Mock-only ingest tests that never verify shared table contract fields (`run_id`, status vocabulary).
- Gate scripts that check legacy route shapes (`/api/cron`) instead of architecture policy.
- Freshness tests that verify one date field but ignore mixed-vintage semantics.

## Fitness-to-Purpose Scenarios

### Scenario 1: Ingest Telemetry Contract Split

**Requirement tag:** [Req: inferred — REQ-001]

**What happened:** `ops.ingest_run` requires `run_id`, but legacy SQL writers insert rows without it while TS helpers write uppercase status enums. This permits partial observability and fragmented status semantics across the same table.

**The requirement:** All ingest writers must use one lifecycle contract: explicit `run_id`, canonical status vocabulary, and deterministic start/success/failure transitions.

**How to verify:** `test_req001_all_ingest_writers_include_run_id_and_canonical_status` in `quality/test_functional.py`.

### Scenario 2: Forbidden Cron Surface Drift

**Requirement tag:** [Req: inferred — REQ-002]

**What happened:** Architecture policy forbids Vercel cron routes, but middleware and verification logic still encode `/api/cron` assumptions. This can block compliant builds or push engineers toward forbidden route resurrection.

**The requirement:** Auth middleware and verification gates must validate pg_cron-only architecture without requiring `/api/cron` endpoints.

**How to verify:** `test_req002_no_runtime_or_gate_dependency_on_api_cron_path` in `quality/test_functional.py`.

### Scenario 3: Intraday Fallback Promise Divergence

**Requirement tag:** [Req: inferred — REQ-003]

**What happened:** `/api/zl/intraday` claims a 15m→1m fallback in comments while querying only `mkt.price_15m`. Consumers can trust resilience that is not implemented.

**The requirement:** Either implement fallback behavior to `mkt.price_1m` or remove fallback claims and align client assumptions.

**How to verify:** `test_req003_intraday_route_contract_matches_implemented_fallback` in `quality/test_functional.py`.

### Scenario 4: Envelope Contract Drift Across Surfaces

**Requirement tag:** [Req: inferred — REQ-004]

**What happened:** Route responses, shared API contract types, and parity gate checks disagree about optional vs required envelope fields (`warning`, shape differences). False-negative parity failures can mask real regressions.

**The requirement:** Contract types, route envelopes, and verification scripts must share one consistent envelope policy.

**How to verify:** `test_req004_contract_and_gate_align_on_warning_semantics` in `quality/test_functional.py`.

### Scenario 5: Mixed-Vintage Freshness Misread

**Requirement tag:** [Req: inferred — REQ-005]

**What happened:** Risk-factor API sets canonical `as_of_date` to oldest component date while also exposing min/max range. Dashboard messaging can be interpreted as fully stale even when only part of vector is stale.

**The requirement:** Mixed-vintage data must be explicitly signaled as degraded with field-level transparency.

**How to verify:** `test_req005_risk_factor_payload_exposes_mixed_vintage_metadata` in `quality/test_functional.py`.

### Scenario 6: Forecast Horizon Normalization Drift

**Requirement tag:** [Req: inferred — REQ-006]

**What happened:** `normalizeHorizon` logic is duplicated across three endpoints. A one-file edit can desynchronize forecast, target-zone, and probability surfaces.

**The requirement:** Horizon normalization constants and mapping behavior must remain equivalent across all forecast-facing endpoints.

**How to verify:** `test_req006_normalize_horizon_logic_is_identical_across_three_routes` in `quality/test_functional.py`.

### Scenario 7: Snapshot Source Whitelist Regression

**Requirement tag:** [Req: inferred — REQ-004]

**What happened:** Snapshot source labels are hard-whitelisted. A source-label drift can silently null out snapshots and trigger unannounced fallback narratives.

**The requirement:** Whitelist changes must be deliberate, and fallback behavior must expose reliability downgrade cues.

**How to verify:** `test_scenario_snapshot_source_whitelist_is_explicit_and_auditable` in `quality/test_functional.py`.

### Scenario 8: Scaffold Pipeline Misreported as Ready

**Requirement tag:** [Req: inferred — REQ-001]

**What happened:** `pipeline.py --all` includes scaffold phases returning `status: scaffold`. Operators can read full-run completion as inference readiness.

**The requirement:** Scaffold status must remain explicit and blocked from “ready” interpretation until implemented.

**How to verify:** `test_scenario_pipeline_order_and_scaffold_modules_are_visible` in `quality/test_functional.py`.

## AI Session Quality Discipline

1. Read `quality/QUALITY.md` and `quality/REQUIREMENTS.md` before changing behavior.
2. Preserve every REQ `Pattern` tag in both Markdown and manifest outputs.
3. Treat cross-file consistency requirements as first-class; do not verify routes in isolation.
4. Run mechanical enumeration verification before asserting completeness claims.
5. Never classify a payload as fresh without validating mixed-vintage metadata fields.
6. Do not close a quality phase until artifacts, manifests, and receipts are all on disk.

## The Human Gate

- Procurement risk appetite decisions for stale or mixed-vintage data tolerance.
- Final severity of architecture-policy drift findings that affect release timelines.
- Decisions to standardize envelope shape vs retaining backward-compatible variants.
- Approval to change ingest status vocabulary in production tables.
- Go/no-go on interpreting scaffold pipeline phases as launch-ready.
