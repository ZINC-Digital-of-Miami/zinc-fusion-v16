## Summary
`ops.ingest_run.status` is written with incompatible literal sets across SQL jobs and TypeScript helpers, so operational status semantics diverge by writer.

## Spec reference
- Requirement: REQ-003
- Spec basis: REQ-003 requires all `ops.ingest_run` writers to use one canonical status vocabulary.
- Behavioral contract quote: "all `ops.ingest_run` writers to use one canonical status vocabulary"

## The code
At `lib/server/ingest-run.ts:3-4` and SQL writer locations including `supabase/migrations/20260414001_ingest_zl_intraday.sql:51-52,139-141` and `supabase/migrations/202605080003_supabase_trusted_site_fill_hourly.sql:56,365,382`, SQL and TS writers persist different status literals.

## Observable consequence
Grepping writer literals across the cited SQL migrations and TypeScript helper reproduces a split vocabulary (`running/ok/failed/error` vs `RUNNING/SUCCESS/FAILED`). Expected behavior is one canonical status set across producers; actual behavior requires producer-specific interpretation and breaks uniform telemetry semantics.
